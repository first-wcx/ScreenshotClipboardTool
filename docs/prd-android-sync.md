# PRD：Android 适配 + 多设备同步传输

## 项目信息

- **Language**: 中文
- **Programming Language**: Python (Kivy + Buildozer 打包 Android APK) / 后端同步服务 Python
- **Project Name**: `integrated_capture_clipboard_android_sync`
- **原始需求复述**：将截图剪切板工具（当前支持 Windows/macOS/Linux）移植到 Android 平台，并实现 Windows/macOS/Linux/Android 之间剪贴板内容的互传同步（文本、图片、文件）。

---

## 产品定义

### 产品目标

1. **Android 可用**：将截图 + 剪贴板历史的核心体验带到 Android，让移动端用户也能快速截屏、编辑、管理剪贴板内容
2. **跨设备互通**：Windows/macOS/Linux/Android 四端剪贴板内容实时同步，实现"复制即同步、粘贴即使用"的无缝体验
3. **安全可控**：同步传输端到端加密、局域网优先、用户完全掌控数据流向

### 成功指标

| 指标 | 目标 |
|------|------|
| Android 端核心功能可用率 | 截图、剪贴板历史、同步三项核心功能 P0 需求 100% 实现 |
| 跨设备文本同步延迟 | 局域网 < 500ms，云端中转 < 3s |
| Android 端 APK 体积 | < 30MB |
| 同步成功率 | 文本 > 99%，图片 > 95% |

### 用户故事

1. **作为办公用户**，我想在电脑上复制一段文字后直接在手机上粘贴，这样不用手动输入或通过微信转发
2. **作为设计师**，我想在电脑上截图后直接在 Android 设备上查看和编辑，这样可以在移动端继续处理
3. **作为移动用户**，我想在 Android 手机上截屏并编辑标注，然后把图片同步到电脑使用
4. **作为多设备用户**，我想通过简单的配对流程将手机和电脑连接，而不需要手动输入 IP 地址和端口
5. **作为隐私敏感用户**，我只想在局域网内同步数据，不想让剪贴板内容经过任何第三方服务器

### 关键交互流程

#### 截屏流程

```
用户点击"截屏" → 请求 MediaProjection 权限
  → 获取屏幕画面 → 进入截图编辑器
  → 编辑完成 → 选择操作：
    ├── 保存到本地
    ├── 复制到剪贴板（自动触发同步）
    └── 分享（系统分享面板）
```

#### 配对流程

```
方案 A - 扫码配对（推荐）：
  Android 端点击"添加设备" → 扫描二维码
  → 桌面端显示二维码（含 IP + 端口 + 临时 Token）
  → Android 扫码后自动连接 → 桌面端确认 → 配对完成

方案 B - PIN 码配对：
  Android 端点击"添加设备" → 手动输入桌面端 IP
  → 桌面端弹出 PIN 码 → Android 输入 PIN → 配对完成
```

### 与桌面端的关键差异

| 特性 | 桌面端 | Android 端 |
|------|--------|-----------|
| 全局热键 | Alt+A/S/D, Ctrl+Shift+M | 不可用；替代：通知栏快捷按钮 + 浮动按钮 |
| 系统托盘 | pystray 托盘图标 | 不可用；替代：前台 Service + 通知栏常驻 |
| 截屏方式 | ImageGrab.grab() 全屏/区域选择 | MediaProjection API，需用户授权 |
| 区域截图 | 自由框选区域 | 截全屏后裁剪（Android 限制） |
| 贴图 | 窗口置顶 FloatingCaptureWindow | 需要 SYSTEM_ALERT_WINDOW 权限，悬浮窗实现 |
| 剪贴板监听 | 轮询 clipboard sequence | ClipboardManager.OnPrimaryClipChangedListener |
| 文件系统 | 直接访问路径 | SAF/ContentProvider，受限存储 |
| GUI 框架 | Tkinter | Kivy / 原生 Android View |
| 后台运行 | 最小化到托盘 | 前台 Service（需通知栏） |

## 多设备同步设计

### 连接/配对流程

#### 设备发现

```
┌─────────┐    mDNS 广播     ┌─────────┐
│ Android │ ◄──────────────► │ Desktop │
│  端     │   _icc_sync._tcp │  端     │
└─────────┘                  └─────────┘
```

1. 桌面端启动后广播 mDNS 服务（`_icc_sync._tcp`），包含设备名、端口
2. Android 端扫描同网络 mDNS，发现可用设备
3. 如果 mDNS 不可用（如跨网段），退化为手动输入 IP 地址

#### 配对握手

```
Android                     Desktop
  │                           │
  │──── CONNECT(ip, port) ───►│
  │                           │
  │◄─── CHALLENGE(nonce) ─────│
  │                           │
  │──── RESPONSE(HMAC) ──────►│  ← HMAC(secret, nonce)
  │                           │
  │◄─── ACK + device_info ────│
  │                           │
  │     [配对完成，进入同步]    │
```

- 首次配对通过扫码获取临时 Token，后续使用 shared_secret 做 HMAC 认证
- 配对信息本地持久化，重连无需重复配对

### 传输方式

#### 局域网直连（P0）

- **协议**：TCP 长连接 + JSON 消息帧（兼容现有 `ClipboardSyncManager` 协议）
- **端口**：默认 8765（复用现有配置）
- **消息格式**：延续现有 `\n` 分隔 JSON 行协议，图片 base64 编码
- **文件传输**：延续现有流式传输机制（header + raw bytes）
- **优势**：零配置、低延迟、无第三方依赖
- **局限**：需同一局域网

#### 云端中转（P2）

- **方案**：WebRTC DataChannel + TURN 服务器
- **信令**：通过轻量信令服务器交换 SDP/ICE
- **适用**：跨网络（如家里电脑 ↔ 公司手机）
- **数据路径**：优先 P2P 直连，不可达时走 TURN 中继

### 同步策略

| 策略 | 说明 | 触发方式 |
|------|------|---------|
| 实时推送 | 剪贴板内容变化时立即推送到所有已配对设备 | 自动（监听剪贴板 onChange） |
| 去重过滤 | 基于 digest 判断内容是否已同步，避免回环 | 自动 |
| 离线缓存 | 目标设备离线时，本地缓存最近 N 条变更，重连后批量推送 | 自动 |
| 手动推送 | 用户选择某条历史记录，手动推送到指定设备 | 用户触发 |

**回环防护**：
- 每条同步消息携带 `origin` node_id
- 接收端不再广播自己收到的内容
- digest 去重：已存在相同 digest 的记录则跳过

### 安全性考虑

| 威胁 | 对策 |
|------|------|
| 未授权设备连接 | 共享密钥 HMAC 认证 + 首次配对需扫码/PIN 确认 |
| 局域网窃听 | 传输层可选 TLS 加密（P1），端到端 AES-256-GCM（P2） |
| 剪贴板敏感数据泄露 | 用户可选择关闭文本/图片同步；隐私模式下暂停同步 |
| 设备仿冒 | 配对时交换设备证书，重连时验证 |
| 大量数据消耗 | 图片同步体积限制（≤32MB）；用户可关闭图片/文件同步 |

## 待确认问题

1. **Android 技术栈选择**：使用 Kivy (Python) 保持代码复用 vs 使用 Kotlin/Swift 原生开发更好的用户体验？Kivy 适合快速移植但 UI 体验可能不如原生，原生开发需要重写大量逻辑
2. **Android 最低版本**：支持 Android 8.0+ (API 26) 还是 Android 10+ (API 29)？更高的最低版本可以简化存储权限和 MediaProjection 的处理
3. **同步协议兼容性**：Android 端是否完全复用现有 TCP+JSON 行协议？还是趁机升级为 WebSocket/Protobuf 等更通用的协议？
4. **云端中转方案**：是否需要云端同步？如果需要，是否自建信令服务器还是使用第三方服务（如 Firebase Realtime Database）？
5. **贴图功能**：Android 悬浮窗需要 `SYSTEM_ALERT_WINDOW` 权限（用户需手动授权），是否作为 P0 功能还是降级为 P1？
6. **Android 文件同步范围**：Android 存储受限（Scoped Storage），文件同步的来源和目标范围如何界定？是否仅限应用私有目录？
7. **多设备上限**：一个设备最多同时与几台设备保持同步连接？建议 5 台，是否合理？
8. **品牌与名称**：Android 端应用名称和图标是否沿用桌面端"截图剪切板工具"，还是单独命名？

## 待确认问题（已决策）

1. **Android 技术栈选择**：✅ **Kotlin 原生** — 最佳用户体验，需要用 Kotlin 重写 Android 端逻辑
2. **Android 最低版本**：✅ **Android 10+ (API 29)** — 简化 Scoped Storage 和 MediaProjection 处理
3. **同步协议兼容性**：✅ **升级为 WebSocket** — 更通用，方便后续 Web 端扩展，桌面端需同步改造
4. **云端中转方案**：✅ **自建中转服务器** — 完全掌控数据，Python 实现 WebSocket 信令+TURN 中继
5. **贴图功能**：✅ **P1** — 需要 SYSTEM_ALERT_WINDOW 权限，不作为 MVP 必须
6. **Android 文件同步范围**：✅ **应用私有目录 + SAF 选择的文件** — Scoped Storage 下限制范围
7. **多设备上限**：✅ **5 台** — 同时连接上限 5 设备
8. **品牌与名称**：✅ **沿用"截图剪切板工具"** — Android 端同一品牌，统一图标风格
